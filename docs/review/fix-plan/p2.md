## Scope

Medium-severity type-system and dispatch risks that become severe with scale/modules:

- Method ambiguity resolution is not explicit/deterministic.
- Duplicate impl handling relies on downstream Go compile errors.
- Trait constraints are tracked globally, not as first-class obligations.
- Monomorphization cache is present but not yet identity-stable for modules.

## Concrete Work

- In `/Users/zlw/src/marmoset/marmoset-ml/lib/frontend/typecheck/trait_registry.ml`:
  - Reject duplicate `(trait, type)` impls at registration.
  - On method lookup, if more than one trait impl provides same method name for receiver type, return ambiguity error (do not pick first).

2. Move constraints toward explicit obligations.
- Define an internal obligation type in type inference:
  - `type obligation = { trait_name: string; typ: mono_type; reason: ... }`
- Inference should collect obligations from:
  - constrained generics,
  - method calls through constrained vars,
  - operator desugaring hooks (when enabled).
- Solve obligations at call/generalization boundaries with explicit diagnostics.

3. Replace global constraint leakage.
- Remove or tightly scope global constraint store in `/Users/zlw/src/marmoset/marmoset-ml/lib/frontend/typecheck/infer.ml`.
- Ensure per-program/per-check isolation (mandatory for REPL and future LSP server process).

4. Strengthen monomorphization instantiation identity for modules.
- Keep current dedupe set, but evolve key shape from `(func_name, concrete_types)` to symbol identity:
  - `(module_path, function_name, arity, normalized_type_args, dispatch_mode)`
- Add a design hook for trait-related instantiations if/when trait-object/existential paths are added.
- Keep output deterministic (stable ordering by key).

## Regression Tests

1. Ambiguous method name across traits should fail with clear error.
2. Duplicate impl for same `(trait, type)` should fail during typecheck, not Go compile.
3. REPL multi-input sessions should not leak constraints across independent definitions.
4. Instantiation keys remain collision-free when same function names exist in different module scopes (test scaffolding can be added before full modules ship).

## Exit Criteria

- No implicit “first match wins” method dispatch.
- No duplicate impls accepted silently.
- Constraint handling is scoped, explicit, and diagnosable.
- Instantiation identity is ready for module namespacing.
