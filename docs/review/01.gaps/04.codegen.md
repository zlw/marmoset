D. Backend / codegen (Go)

D1) Monomorphization cache keyed by (fn, typeArgs) is essential

You asked earlier “what do you mean by cache keyed by (fn, typeArgs)”. It’s this:

When you compile:

fn f[a](x: a) -> a = x
f(1)
f(2)

You don’t want to emit f_int twice.

Future issue: without it, codegen duplicates explode, build times balloon, and you get weird name collisions.

Fix: maintain:
	•	instantiations: map<(fn_id, type_subst), emitted_name>
	•	during emission, look up before generating

Same for trait method instantiations.

⸻

D2) Trait object encoding needs a stable ABI, especially for modules/FFI

Right now it’s:
	•	Go interface with getters/methods
	•	auto-generate getters on concrete types

Future issues:
	•	adding modules: methods on types may not be addable across Go packages
	•	FFI: you might want to pass Marmoset values to Go and back; interface-based erasure may cause allocations/escapes

Fix options:
	•	keep everything in one Go package initially (easiest)
	•	once modules exist, decide:
	•	compile whole project into one Go package (pragmatic, many languages do this)
	•	or multiple Go packages (harder; affects trait object and “impl” generation)

Given your goals, compile to one Go package per build is the pragmatic choice.

⸻

D3) Records codegen must use canonical shapes (again)

This is both type system + codegen.
Also: anonymous records should reuse canonical type when shape known.

⸻

D4) Spread update codegen can be optimized without “compiler optimizations”

Even without heavy optimization passes, you can do a lot in emitter:
	•	if you emit canonical struct types, spread { ...p, x: 5 } can become:
	•	p2 := p; p2.X = 5 (mutation) only if it doesn’t violate purity/aliasing semantics
	•	if the base record is known not to alias, you can do copy-on-write safely

Future issue: naive “rebuild struct literal with all fields” can bloat code and allocations.

Fix: introduce an internal “record update” lowering that emits a copy-then-override pattern.

⸻

D5) Equality/ord/hash for records and collections should be derived strategically

If you derive eq for records by field-wise comparison, fine.
But if you want performance later, you’ll need:
	•	short-circuit logic
	•	stable field order
	•	for hash, a combiner function that’s stable across runs (important for test determinism)
