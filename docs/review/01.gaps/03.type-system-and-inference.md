C. Type system & inference

C1) Trait solver vs HM inference interaction needs a defined “obligation” mechanism

You have HM + bidirectional + traits with constraints. Right now your plan checks constraints “at call sites” using get_generic_constraints func_expr, which is too weak.

Future issue: you’ll need trait obligations from many places:
	•	operator desugaring
	•	dot method resolution
	•	derive constraints for generic types
	•	record field constraints via traits/rows
	•	typeclass-style functions show(x) needing a: show

Fix: introduce a first-class internal concept:
	•	obligation = type implements trait
	•	inference produces a list of obligations alongside type
	•	obligations are solved at generalization boundaries / call sites depending on design
Even if you keep it simple initially, have the structure.

⸻

C2) Ambiguity handling needs user-facing rules

Your solver has Ambiguous of impl_info list and tries “most specific”.

Future issue: “most specific” is non-trivial with row constraints, type aliases, and modules. You’ll ship weird behavior unless you define coherence rules.

Fix: define a strict rule:
	•	if multiple impls match, error unless one is strictly more specific by a clear syntactic criterion (e.g., fewer type vars, more concrete head type)
	•	in v1, probably just error on ambiguity (simpler + more predictable)

⸻

C3) Default trait methods + dispatch must be specified

Your trait methods can have default bodies. Where do those live?
	•	do they desugar into generated impls?
	•	can they call other trait methods (requires self constraints)?
	•	do defaults participate in monomorphization?

Future issue: tricky bugs or missing features.

Fix: simplest implementation:
	•	treat default methods as generated functions parameterized by the trait dictionary (even if you don’t surface dictionaries)
	•	or, during trait registration, synthesize a “default impl template” and copy into impl block if method missing.

⸻

C4) Existentials and trait objects are already “half present”

list[named] is existential packaging. But you don’t have explicit existentials yet, and module boundaries will stress representation.

Future issue: you’ll later add real existentials and discover your trait object encoding is incompatible.

Fix: treat trait-object encoding as a first-class internal representation now:
	•	trait_object(named) is not “just a type”
	•	record how it erases type params
	•	define equality/hash/show for trait objects (likely not implementable generally)
